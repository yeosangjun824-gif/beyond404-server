# LG SwapIt 지금까지 진행한 개발 기록

이 문서는 구현 과정 기록 규칙을 만들기 전에 진행했던 내용을 한 번에 정리한 기록입니다.

앞으로는 기능을 구현하고 push할 때마다 별도의 과정 기록 문서를 남기기로 했습니다. 이 문서는 그 이전까지의 기획, 설계, 구현, 시행착오를 한 번에 모아둔 기준 문서입니다.

## 1. 프로젝트 출발점

처음에는 LG SwapIt 서비스 아이디어를 앱 프로토타입 이미지로 공유하고, 전체 서비스를 어떻게 구현할지 논의했습니다.

핵심 서비스 흐름은 다음과 같이 정리했습니다.

```txt
1. VLM 기반 스마트 촬영
2. 사진 기반 예상 보상가 안내
3. 수거 및 설치 예약
4. 수거 크루 실시간 트래킹
5. 최종 검수 후 크레딧 발급
6. 자원 순환 결과 확인
```

초기 논의에서 가장 중요하게 정한 방향은 사진만으로 최종 가치를 확정하지 않고, 먼저 예상 보상가 범위를 안내한 뒤 수거 후 해체/검수를 통해 최종 크레딧을 확정하는 방식이었습니다.

## 2. 기술 스택 결정

개발 기간이 2~3주 정도이고, 팀원이 개발 기초 지식이 많지 않은 상황을 고려해 빠르게 구현할 수 있는 구조를 선택했습니다.

최종 기술 스택은 다음과 같이 정했습니다.

```txt
Frontend: Next.js + TypeScript + Tailwind CSS + TanStack Query
Backend: Java 21 + Spring Boot + Spring Web + Spring Data JPA + Validation + Lombok + Springdoc OpenAPI + Actuator
DB: H2(local) -> PostgreSQL(deploy)
Storage: AWS S3 SDK
Deploy: AWS Amplify(front) + AWS App Runner(back)
```

선택 이유는 다음과 같습니다.

- Next.js: 고객용 웹앱/PWA를 빠르게 만들기 위해 사용
- TypeScript: 프론트 코드의 타입 오류를 줄이기 위해 사용
- Tailwind CSS: 모바일 화면 UI를 빠르게 구성하기 위해 사용
- TanStack Query: 프론트에서 API 요청 상태를 관리하기 위해 사용
- Spring Boot: 백엔드 API 서버를 안정적으로 만들기 위해 사용
- Spring Web: REST API를 만들기 위해 사용
- Spring Data JPA: DB 저장/조회 코드를 쉽게 만들기 위해 사용
- Validation: API 요청값 검증을 위해 사용
- Lombok: Java 반복 코드를 줄이기 위해 사용
- Springdoc OpenAPI: Swagger API 문서화를 위해 사용 예정
- Actuator: 서버 상태 확인을 위해 사용
- H2: 로컬 개발용 임시 DB
- PostgreSQL: 배포 환경에서 사용할 실제 DB
- S3: 가전 사진 저장용
- Amplify: 프론트 배포용
- App Runner: 백엔드 배포용

## 3. Repository 구성 결정

초기에는 하나의 프로젝트 폴더에서 프론트와 백엔드를 모두 구성하는 방식을 논의했지만, GitHub repository를 분리하는 방식으로 정리했습니다.

최종 repo 구조는 다음과 같습니다.

```txt
beyond404-web     고객용 ThinQ 앱 프론트엔드
beyond404-server  백엔드 API 서버
beyond404-crew    수거 크루용 앱 프론트엔드
```

추후 팀원이 프론트를 고치고 사용자는 백엔드를 고치는 상황을 고려해, 프론트와 백엔드 작업 영역을 명확하게 나누기로 했습니다.

현재 약속:

- `beyond404-web`은 팀원이 프론트 작업 중이므로 Codex가 건드리지 않음
- 사용자가 명시적으로 `push 해줘`라고 할 때만 push
- 백엔드 작업은 `beyond404-server` 기준으로 진행
- 백엔드 작업 기록은 `beyond404-server/docs`에 남김

## 4. 폴더 구조 설계

초기 설계에서 프론트, 백엔드, DB, 문서, 인프라 폴더를 어떻게 구성할지 정리했습니다.

프론트는 기능별로 다음 영역을 나누었습니다.

```txt
capture           가전 촬영
inspection        VLM 분석 결과
pre-valuation     예상 보상가
booking           수거 예약
tracking          수거 진행 상태
final-valuation   최종 검수 결과
credit            크레딧
recycling         자원 순환 리포트
```

백엔드는 레이어드 아키텍처 기준으로 다음 영역을 나누었습니다.

```txt
controller    프론트 요청을 받는 REST API 계층
service       비즈니스 로직 계층
repository    DB 조회/저장 계층
domain        Entity, Enum 등 핵심 도메인 모델
dto           Request/Response 데이터 전달 객체
integration   VLM, S3 등 외부 서비스 연동
config        CORS, Swagger 등 설정
common        공통 응답, 예외 처리, 유틸
```

DB 설계 문서와 공유용 SQL은 별도 `database` 폴더를 두는 방향이 좋다고 판단했습니다.


`CUSTOMER_API_SPEC.md`에는 고객용 앱과 백엔드가 주고받을 API 명세를 정리했습니다. 현재 구현된 API와 앞으로 Google Maps, VLM 모델, 크레딧, 마켓까지 확장할 API를 나누어 작성했습니다.

## 7. 고객용 앱 프론트 프로토타입 구현

고객용 앱은 ThinQ 앱 안에 들어가는 하나의 기능처럼 보이도록 구성했습니다.

주요 구현 흐름:

```txt
ThinQ 홈 화면
-> LG SwapIt 기능 진입
-> 가전 선택
-> 가전 촬영
-> Mock VLM 인식
-> 가전 정보 확인/수정
-> 감정 중 화면
-> 예상 보상가
-> 예약 또는 바로 콜
-> 수거 트래킹
-> 최종 검수 중
-> 최종 감정가 확인
-> 크레딧 받기
-> LG 마켓으로 이동
```

UI 관련 시행착오:

- 처음에는 iPhone 프레임과 앱 진입 액션이 자연스럽지 않아 여러 번 수정했습니다.
- 앱 첫 화면을 단순 카드형이 아니라 LG 색상과 인도 문양이 들어간 표지 화면처럼 바꾸었습니다.
- 인도 문양은 여러 시안을 거쳐 LG 색상 배경 위에 은은하게 보이도록 조정했습니다.
- 카메라 화면은 파일 업로드가 아니라 실시간 카메라 촬영 느낌으로 바꾸었습니다.
- 가전 종류에 따라 촬영 프레임 크기를 다르게 두기로 했습니다.
- 사진 촬영 후 바로 감정으로 가지 않고, Mock VLM이 인식한 가전 정보를 고객이 확인/수정한 뒤 감정 단계로 넘어가도록 정리했습니다.

## 8. 고객용 앱 카메라 촬영 흐름

최종적으로 정리한 촬영 흐름은 다음과 같습니다.

```txt
가전 선택
-> 사진 등록으로 이동
-> 카메라 화면 진입
-> 실시간 카메라 미리보기
-> 가전별 프레임 맞추기
-> 촬영
-> 촬영 결과 확인
-> Mock VLM 정보 확인/수정
-> 감정 중
-> 예상 보상가
```

카메라 관련 시행착오:

- 처음에는 사진 업로드처럼 보였고, 사용자는 실제 카메라 촬영을 원했습니다.
- 카메라 화면에서 프레임과 안내문이 겹치는 문제가 있었습니다.
- 대기 화면의 데모 배경 요소가 초록 프레임과 겹쳐 보였고, 이를 단순한 그리드 배경으로 정리했습니다.
- PC 카메라에서 사람 외 배경이 흐리게 보이는 문제가 있었는데, 이는 앱 CSS가 아니라 OS/브라우저 카메라 효과일 가능성이 높다고 판단했습니다.
- 코드에서는 후면 카메라 우선 요청, `backgroundBlur: false` 요청, `filter:none` 처리 등을 추가했습니다.
- 모바일 카메라 테스트를 위해 HTTPS localtunnel 주소를 만들어 사용했습니다.
- 촬영 결과 확인 화면에서 실제 촬영 사진이 보이지 않는 문제가 있어, 캡처 이미지를 `dataURL`로 즉시 저장하고 결과 화면에 표시하도록 수정했습니다.

## 9. 고객용 앱 예약 / 바로 콜 / 트래킹 흐름

예약 단계는 두 가지 방식으로 정리했습니다.

```txt
1. 예약 수거
   - 달력에서 날짜 선택
   - 시간 선택
   - 수거 위치 설정
   - 예약 완료 후 ThinQ 홈으로 복귀
   - 홈에 예약 시간이 표시됨

2. 바로 콜
   - 지도에서 수거 위치 선택
   - 상세 주소 입력
   - 근처 수거 크루 호출
   - 기사 찾는 중 화면
   - 매칭 성공 후 기사 위치 트래킹
```

데모를 위해 예약 완료 후 10초 뒤 수거 완료, 최종 검수 중 10초 뒤 검수 완료 같은 자동 상태 전환도 구성했습니다.

트래킹 관련 추가 논의:

- Google Maps는 비용이 발생할 수 있어 무료 대안도 검토했습니다.
- 주소 자동완성은 무료 API 또는 백엔드 프록시 방식으로 구현하는 방향을 논의했습니다.
- 실시간 위치 트래킹은 Firebase Firestore 기반 구조도 별도로 검토했습니다.
- 이후 기존 Next.js 고객용 앱 기술 스택에 맞춰 구현해야 한다고 정리했습니다.

## 10. 최종 검수 / 재검수 / 크레딧 흐름

수거 후에는 바로 크레딧을 확정하지 않고, 최종 검수 중 상태를 거치도록 했습니다.

정리한 흐름:

```txt
수거 완료
-> ThinQ 홈에 "최종 검수 중" 표시
-> 검수 완료 알림
-> 고객이 최종 감정가 확인
-> 감정 사유 확인
-> 크레딧 받기 또는 재검수 요청
```

재검수는 한 번만 가능하도록 정리했습니다.

재검수 후에는 다시 재검수 요청 버튼을 보여주지 않고, `크레딧 받기`만 보이도록 수정했습니다.

크레딧을 받은 뒤에는 다음 버튼을 두기로 했습니다.

```txt
ThinQ 홈으로 돌아가기
크레딧 쓰러가기
```

`크레딧 쓰러가기`는 별도의 LG 마켓 페이지로 이동하도록 정리했습니다.

## 11. LG 마켓 데모

크레딧을 사용할 수 있는 LG 가전 마켓 화면을 데모로 구성했습니다.

논의한 내용:

- 실제 LG 상품 정보, 사진, 가격을 가져오고 싶었지만 무료/라이선스/크롤링 이슈가 있어 데모 상품으로 우선 구성했습니다.
- 상품을 선택하면 상세 설명과 결제하기 버튼이 나오도록 했습니다.
- 크레딧 적용 가격이 보이도록 했습니다.

이 부분은 추후 실제 상품 API 또는 관리자가 입력한 상품 DB로 대체할 수 있습니다.

## 12. 수거 크루 앱 프로토타입

고객용 앱과 별도로 수거 크루가 사용하는 앱도 필요하다고 판단했습니다.

크루 앱 방향:

```txt
로그인
신규 가입
휴대폰 본인인증
가입 정보 입력
주소/수거 가능 지역 입력
약관 동의
정산 계좌 등록
가입 완료
콜 목록 확인
수거 요청 수락
수거 상태 변경
```

기술 스택은 고객용 프론트와 동일하게 가져가기로 했습니다.

백엔드는 별도 크루 서버를 만들기보다, 하나의 `beyond404-server` 안에 크루 관련 도메인과 API를 추가하는 방식이 좋다고 정리했습니다.

## 13. 백엔드 현재 구현 상태

현재 백엔드는 Spring Boot 기반 Mock API 서버입니다.

현재 구현된 API:

```txt
POST /api/swap-requests
GET  /api/swap-requests/{id}
POST /api/swap-requests/{id}/photos
POST /api/swap-requests/{id}/booking
POST /api/swap-requests/{id}/final-valuation/mock
```

현재 저장 방식:

```txt
DB 미연결
ConcurrentHashMap 기반 메모리 저장
```

현재 백엔드 역할:

- 교환 신청 생성
- 사진 분석 Mock 처리
- 예상 보상가 Mock 응답
- 예약 확정 Mock 처리
- 최종 감정가/크레딧 Mock 처리

아직 구현되지 않은 백엔드 기능:

- 실제 DB 저장
- 이미지 업로드/S3 연동
- VLM API 연동
- 예약 가능 날짜/시간 조회
- 바로 콜 매칭
- 크루 위치 트래킹
- 재검수 요청
- 알림
- 크레딧 지갑
- LG 마켓 상품/주문

## 14. 고객용 앱 API 명세서 작성

고객용 앱 프론트와 백엔드 연결을 명확히 하기 위해 `CUSTOMER_API_SPEC.md`를 작성했습니다.

명세서에는 다음 내용을 포함했습니다.

```txt
1. 기본 규칙
2. 고객용 앱 전체 흐름
3. 현재 구현된 API
4. 고객용 MVP 구현 예정 API
5. 바로 콜 / 기사 호출 API
6. 트래킹 API
7. Google Maps 연동 예정 API
8. VLM 모델 연동 예정 API
9. 최종 검수 / 재검수 API
10. 크레딧 API
11. LG 마켓 API
12. 알림 API
13. 상태값 정의
14. 구현 우선순위
```

Google Maps 연동은 주소 자동완성, 좌표 변환, 예상 도착 시간 계산 API로 나누었습니다.

VLM 연동은 S3 presigned URL 발급, 이미지 업로드 완료 처리, VLM 분석 요청, 재촬영 요청으로 나누었습니다.

## 15. Git / Push 관련 약속

중간에 중요한 협업 규칙을 정했습니다.

```txt
1. 사용자가 명시적으로 "push 해줘"라고 말할 때만 push한다.
2. 팀원이 프론트 작업 중이므로 Codex는 beyond404-web을 건드리지 않는다.
3. 백엔드 작업은 beyond404-server에서 진행한다.
4. 앞으로 push할 때는 구현 과정 문서도 같이 작성한다.
5. 구현 과정 문서에는 우리 대화 내용, 요구사항, 시행착오, 수정 파일, 테스트 결과, 커밋 메시지를 함께 남긴다.
```

프론트 카메라 캡처 개선 작업은 사용자가 명시적으로 push를 요청하여 다음 커밋으로 push했습니다.

```txt
ce8a168 feat: improve appliance camera capture flow
```

이후부터는 프론트 repo는 팀원이 작업하므로 건드리지 않기로 했습니다.

## 16. 주요 시행착오 요약

### 16.1 폴더 위치 혼동

처음에는 `DX_project`, `tlqja`, `DX` 폴더가 섞여 있어 작업 위치가 혼동되었습니다.

최종 기준:

```txt
C:\Users\yeosa\Desktop\DX
```

현재 기준 repo:

```txt
C:\Users\yeosa\Desktop\DX\beyond404-web
C:\Users\yeosa\Desktop\DX\beyond404-server
C:\Users\yeosa\Desktop\DX\beyond404-crew
```

### 16.2 프론트와 백엔드 연결 범위

고객용 앱 화면은 많이 구현되었지만, 백엔드 연결은 아직 일부 Mock API에만 연결된 상태입니다.

### 16.3 모바일 카메라 문제

모바일 브라우저에서 카메라를 사용하려면 HTTPS가 필요하다는 점을 확인했습니다.

임시 해결:

```txt
localtunnel로 https://...loca.lt 주소 생성
```

### 16.4 카메라 배경 흐림 문제

사람 외 배경이 흐리게 보이는 문제는 앱 CSS가 아니라 OS/브라우저 카메라 효과일 가능성이 크다고 판단했습니다.

앱 쪽에서는 가능한 범위 내에서 다음 처리를 했습니다.

- 후면 카메라 우선 요청
- backgroundBlur false 요청
- filter/backdrop blur 제거
- 대체 배경 단순화

### 16.5 같은 파일 동시 수정 위험

프론트 작업에서 팀원과 같은 파일을 동시에 고치면 Git conflict가 날 수 있음을 확인했습니다.

따라서 지금부터 Codex는 `beyond404-web`을 건드리지 않고 백엔드만 작업하기로 했습니다.

## 18. 앞으로의 문서 기록 규칙

앞으로 기능을 구현하고 push할 때마다 다음 형식의 기록 문서를 남깁니다.

```txt

```

문서에 포함할 내용:

```txt

4. 구현 내용
5. 테스트 결과
6. Git 기록
7. 다음 작업
```

이 문서부터는 구현 결과뿐 아니라 사용자와 Codex가 어떤 대화를 통해 방향을 정했는지도 함께 남깁니다.
